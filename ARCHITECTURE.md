## 2. 项目一句话概述

**LLM-Pareto-Controller 是一个面向研究实验的 Python 工程，用于研究“大语言模型如何作为高层闭环控制器，基于 Pareto 搜索状态动态调节 NSGA-II 的搜索行为，从而提升多目标任务分配类问题的优化过程”。**

这里有两个关键词：

- **不是让 LLM 直接替代优化器**；
- **而是让 LLM 观察优化状态后给出控制动作**。

换句话说，真正搜索候选解空间的仍然是进化优化器；LLM 的角色是“观察者 + 调参者 + 高层策略器”。

## 3. 要解决的核心研究问题

这个项目面向的不是单纯的“求一个最优解”，而是更偏向以下研究问题：

### 3.1 在线调参是否能改善多目标搜索过程

经典 NSGA-II 在固定参数下运行时，经常会遇到：

- 前期探索不足；
- 中期多样性下降；
- 后期陷入停滞；
- 可行解比例波动较大；
- 不同问题规模下参数很难一套通吃。

因此，本项目试图通过在线控制，在运行时动态调整：

- `mutation_prob`
- `crossover_prob`
- `repair_prob`
- `eta_c`
- `eta_m`
- `local_search_prob`

从而让同一个求解器在不同搜索阶段采用不同策略。

### 3.2 LLM 是否可以扮演“高层控制器”

在本项目里，LLM 不直接处理大规模组合搜索，而是做三件事：

1. **理解状态**：例如可行率是否偏低、HV 是否停滞、多样性是否恶化；
2. **形成策略**：例如“提高探索”“加强可行性修复”“维持收敛节奏”；
3. **输出动作**：把策略映射为具体的可执行参数。

这使得 LLM 的作用更接近一个具备语言推理能力的“实验调度员”，而不是黑盒求解器。

### 3.3 能否形成可复现、可解释的闭环实验系统

研究型工程不能只会跑，还必须能：

- 保存配置与结果；
- 复现相同实验；
- 比较 baseline / ablation；
- 解释某次控制动作为什么发生；
- 导出多次运行的汇总结果。

本仓库围绕这些要求设计了日志、经验池、实验矩阵和指标导出能力。

## 4. 当前工程能力概览

当前仓库已经超出最早的纯 MVP 骨架，具备以下完整能力：

### 4.1 支持的问题类型

#### A. 通用任务分配问题 `task_assignment`

适合表达“任务必须被分配到资源”的离散组合优化场景，支持：

- 成本矩阵；
- 任务负载；
- 资源容量；
- 兼容性矩阵；
- 时间窗约束；
- 阶段顺序约束。

#### B. 动态武器-目标分配问题 `dwta`

这是一个更具体、更研究导向的问题族，支持两种输入方式：

- **实体定义输入**：弹药类型、武器、目标；
- **预计算矩阵输入**：兼容性、杀伤率、需求伤害等矩阵。

### 4.2 支持的控制模式

- `rule`：纯规则控制，不依赖任何 LLM；
- `mock_llm`：模拟 LLM 分析和策略输出，适合离线调试；
- `real_llm`：通过统一客户端接入真实大模型 API。

### 4.3 支持的实验能力

- 单次闭环运行；
- baseline 运行器；
- matched 实验；
- ablation 实验矩阵；
- 指标导出与聚合。

### 4.4 支持的结果与指标

- Hypervolume（HV）
- IGD / IGD+
- Spread
- Spacing
- 可行率、约束违反、多样性、前沿分离等过程指标

## 5. 设计理念：为什么这样分层

本项目采用“问题层—优化器层—状态感知层—控制层—记忆层—基础设施层—实验层”的结构，原因有三：

### 5.1 保证可替换性

如果以后你想：

- 把 `task_assignment` 换成新的组合优化问题；
- 把 NSGA-II 换成 MOEA/D 或自定义进化器；
- 把规则控制换成真正的多角色 LLM 控制链；
- 把 JSONL 存储换成数据库；

那么当前这种分层能尽量减少连带修改。

### 5.2 保证研究对照的公平性

如果优化器和控制器耦合过深，就很难判断“指标提升到底来自控制策略还是来自底层算法偷偷变化”。

因此项目里尽量做到：

- 优化器负责搜索；
- 传感器负责状态抽象；
- 控制器负责动作决策；
- 经验池负责记忆；
- 日志层负责记录事实。

### 5.3 保证工程可维护性

研究代码最常见的问题就是“能跑，但没人敢改”。模块分层的价值是：

- 每个目录职责清楚；
- 测试更容易写；
- 代码定位更快；
- 后续论文实验更容易扩展。

## 6. 顶层目录与职责说明

### 6.1 `problems/`

负责问题建模和解空间定义。

#### `problems/task_assignment/`

处理通用任务分配问题：

- `encoding.py`：解的编码与随机初始化；
- `objectives.py`：成本、不均衡等目标函数；
- `constraints.py`：容量、兼容性、时间窗、阶段约束；
- `repair.py`：对不合法个体做修复。

#### `problems/dwta/`

处理 DWTA 场景：

- `model.py`：数据模型；
- `scenario_builder.py`：由实体配置构建矩阵；
- `precompute.py`：预计算辅助逻辑；
- `encoding.py`：DWTA 决策表示；
- `objectives.py`：DWTA 目标；
- `constraints.py`：弹药容量与兼容性约束；
- `repair.py`：DWTA 修复逻辑。

### 6.2 `optimizers/`

负责底层优化器实现。

#### `optimizers/nsga2/`

- `population.py`：个体结构；
- `operators.py`：交叉、变异、本地搜索等算子；
- `selection.py`：非支配排序、拥挤距离、环境选择；
- `solver.py`：主迭代流程与控制器可调接口。

### 6.3 `sensing/`

负责将“整个种群”压缩成“控制器可读状态”。

- `feasibility_metrics.py`：可行性相关指标；
- `hypervolume.py`：HV 计算接口与实现；
- `pareto_state.py`：构建 `ParetoState`。

### 6.4 `controller/`

负责闭环控制语义与调度。

- `control_semantics.py`：控制状态语义；
- `operator_space.py`：控制器可调的参数空间定义；
- `closed_loop.py`：闭环运行器与控制器接口。

### 6.5 `llm/`

负责 LLM 控制链。

- `analyst.py`：从状态和经验生成诊断；
- `strategist.py`：将诊断映射为高层策略；
- `actuator.py`：将策略转为可执行参数动作；
- `prompts/`：提示词模板。

### 6.6 `memory/`

负责经验池：

- 记录 `state -> action -> reward -> next_state`；
- 支持窗口裁剪与最近经验读取；
- 为后续检索式控制提供基础。

### 6.7 `infra/`

负责底层设施抽象：

- `llm_client.py`：真实 / 模拟 LLM 调用；
- `storage.py`：JSONL 等日志持久化。

### 6.8 `experiments/`

负责研究实验组织，而不是业务逻辑：

- `configs/`：配置文件；
- `baselines/`：基线运行器；
- `ablations/`：消融开关与矩阵；
- `run_matrix.py`：批量运行入口；
- `export_results.py`：结果导出。

### 6.9 `eval/`

负责参考前沿与后处理指标：

- `reference_front.py`
- `metrics.py`

### 6.10 `tests/`

提供从基础模块到主流程的大量测试，保证：

- 问题建模接口不轻易回归；
- 求解器行为相对稳定；
- 闭环控制逻辑可验证；
- 实验脚本和导出链路能运行。

## 7. 一次完整运行的控制流

下面以 `main.py` 驱动的一次运行作为例子。

### 第 1 步：读取配置

`main.py` 会读取 YAML 配置，并通过 Pydantic 模型完成：

- 字段解析；
- 类型校验；
- 约束校验；
- 兼容旧字段名。

这一步的价值是尽可能在“运行前”发现配置错误，而不是在实验跑到中途才崩掉。

### 第 2 步：构建问题实例

根据 `problem.problem_type`：

- 如果是 `task_assignment`，构建标准任务分配输入；
- 如果是 `dwta`，要么用实体配置生成矩阵，要么直接读取预计算矩阵。

### 第 3 步：构建 NSGA-II 求解器

`NSGA2Solver` 负责：

- 初始化种群；
- 评价个体目标与约束；
- 执行交叉、变异、修复；
- 做父代选择和环境选择；
- 在每代结束后给出新的群体。

### 第 4 步：构建状态感知器

`ParetoStateSensor` 从当前种群提取结构化状态，例如：

- `hv`
- `delta_hv`
- `feasible_ratio`
- `rank1_ratio`
- `mean_cv`
- `diversity_score`
- `crowding_entropy`
- `d_dec`
- `d_front`
- `stagnation_len`

### 第 5 步：构建控制器

根据配置模式：

- `rule`：使用启发式规则；
- `mock_llm` / `real_llm`：使用 analyst → strategist → actuator 链。

### 第 6 步：执行闭环运行

`ClosedLoopRunner` 负责推进时间线：

1. 初始化种群；
2. 记录 generation 0 状态；
3. 每代调用求解器推进；
4. 感知新状态；
5. 如果到达控制间隔，则调用控制器；
6. 把动作写回求解器的算子参数；
7. 在下一状态到达后补全经验记录。

### 第 7 步：写出日志与摘要

输出通常包括：

- 状态事件流；
- 控制动作；
- 经验记录；
- 配置快照；
- 运行摘要；
- 可供后处理脚本使用的结构化结果。

## 8. 核心数据流

可以把整个系统理解为下面这条主链路：

```text
配置 YAML
  ↓
问题实例构建
  ↓
NSGA-II 初始化种群
  ↓
种群评价与进化
  ↓
ParetoState 感知
  ↓
经验池读取最近经验
  ↓
控制器生成动作
  ↓
动作写回求解器参数
  ↓
继续进化
  ↓
记录日志、经验与摘要
```

这条链路的关键特点是：

- **状态与动作是显式对象，不是隐藏副作用**；
- **控制动作通过统一参数接口回写**；
- **经验记录形成可追踪的因果片段**。

## 9. 为什么 `ParetoState` 很重要

如果直接把整个人口、全部目标、约束、历史日志原封不动交给控制器：

- 信息量太大；
- 不稳定；
- 难比较不同实验；
- 对 LLM 来说也不利于形成稳定推理模式。

所以项目引入了一个中间层：`ParetoState`。

它的作用相当于“搜索过程的仪表盘”。

### 9.1 典型指标含义

- `hv`：当前 Pareto 前沿整体质量；
- `delta_hv`：与上一个状态相比是否有改进；
- `feasible_ratio`：当前群体中可行解比例；
- `rank1_ratio`：非支配解占比；
- `mean_cv`：平均约束违反度；
- `diversity_score`：目标空间多样性；
- `crowding_entropy`：rank-1 分布均匀性；
- `d_dec`：决策空间多样性；
- `d_front`：rank-1 前沿与被支配解的分离程度；
- `stagnation_len`：停滞持续长度。

### 9.2 这些指标如何帮助控制器

例如：

- `feasible_ratio` 很低，说明需要提升修复或降低过强探索；
- `diversity_score` 很低，说明需要增加变异、降低过度收敛；
- `delta_hv` 很小且 `stagnation_len` 变长，说明搜索可能陷入局部停滞；
- `d_front` 很低，说明当前优劣解混杂，前沿不够清晰。

这就是控制器做在线决策的依据。

## 10. 控制器设计详解

### 10.1 规则控制器的定位

规则控制器不是终点，而是一个非常重要的“可运行基线”。

它的价值在于：

- 不依赖 API key；
- 行为稳定、可解释；
- 能验证闭环数据链路是否正确；
- 可作为 future LLM 控制器的对照组。

### 10.2 LLM 控制链为什么拆成三段

项目中的 LLM 控制链不是“一次 prompt 出所有结果”，而是拆成：

1. `Analyst`：看状态，做诊断；
2. `Strategist`：根据诊断给高层策略；
3. `Actuator`：把策略转成参数动作。

这样设计有几个好处：

- 更贴近人类分析流程；
- 便于消融实验（去掉 strategist / 固定 actuator）；
- 便于替换不同 prompt 模板；
- 便于追踪“错误发生在哪一层”。

### 10.3 为什么需要 capability-aware 参数空间

不同问题和算子并不支持完全相同的参数：

- 某些模式支持 `repair_prob`；
- 某些模式支持 `eta_c` / `eta_m`；
- 某些模式支持 `local_search_prob`；
- 某些模式不支持这些扩展维度。

所以控制器不能“想改什么就改什么”。

当前工程通过 `OperatorCapabilities` 和 `OperatorParams` 显式表达：

- 哪些维度可调；
- 当前值是什么；
- 最终哪些值真正生效。

这能显著减少控制器与求解器之间的隐式耦合。

## 11. 经验池在系统中的意义

经验池不是为了把项目立刻变成强化学习系统，而是为了给闭环控制补上一层“短期记忆”。

每条经验记录包含：

- `state`
- `action`
- `reward`
- `next_state`

### 11.1 它现在解决什么问题

- 让控制器不只看单一时刻，而能参考最近行为；
- 让实验日志具备更完整的因果链；
- 为后续相似状态检索留接口；
- 为 mock / real LLM 提供少量上下文记忆。

### 11.2 它以后可以扩展什么

- 相似状态检索；
- 经验摘要；
- episode 级统计；
- 更精细的 reward 设计；
- 基于经验的策略建议。

## 12. 为什么日志设计成结构化输出

研究实验里，结构化日志比“终端打印好看”重要得多。

当前项目偏向把运行输出保存成 JSONL / JSON / YAML，原因包括：

- 易于后处理；
- 易于聚合多次运行；
- 易于写测试；
- 易于查某一代发生了什么；
- 易于导入分析脚本或 notebook。

### 12.1 常见输出物

- `config_snapshot.yaml`
- `events.jsonl`
- `generation_metrics.jsonl`
- `actions.jsonl`
- `experiences.jsonl`
- `summary.json`

### 12.2 为什么事件流和汇总都需要

- **事件流**适合追踪过程；
- **summary**适合比较结果；
- **配置快照**保证复现；
- **经验日志**补充控制因果链。

## 13. 实验体系如何理解

本仓库不仅支持“跑一次”，还支持“系统性比较”。

### 13.1 baseline

用于比较：

- 标准 NSGA-II；
- 规则控制版本；
- 无经验池版本。

### 13.2 matched experiments

用于保证不同设置在相对可比的条件下运行，比如：

- 相同种子；
- 相同问题规模；
- 相同评估输出结构。

### 13.3 ablation

用于回答“某个模块到底有没有贡献”。

例如：

- `no_memory`
- `no_state_metric_x`
- `fixed_control_interval`
- `no_llm_chain`

### 13.4 export

用于把多次运行结果转换成：

- 汇总表；
- 可报告结果；
- 后续绘图输入。

## 15. 建议的使用方式

### 15.1 如果你只是想先跑起来

按 `README.md` 操作：

1. 创建 Python 3.11 环境；
2. 安装依赖；
3. 运行 `python main.py`；
4. 查看输出目录里的 `summary.json` 和日志文件。

### 15.2 如果你想调试闭环行为

建议顺序：

1. 先跑 `rule`；
2. 再跑 `mock_llm`；
3. 最后才切换 `real_llm`。

这样更容易确认：

- 问题建模是否正确；
- 状态感知是否稳定；
- 控制动作是否合理；
- LLM 接口是否只是“锦上添花”，而不是“引入不确定性来源”。

### 15.3 如果你想做论文式实验

建议至少固定：

- 配置文件版本；
- 运行种子；
- baseline 列表；
- ablation 开关；
- 导出流程；
- 指标统计口径。

## 17. 总结

这个项目的关键价值，不只是“实现了一个 NSGA-II”，也不只是“接了一个 LLM API”。

它真正做的是把以下几件事组织成一个可研究、可复现、可扩展的闭环系统：

- **问题建模**：定义多目标任务分配与 DWTA 场景；
- **优化搜索**：由 NSGA-II 生成和筛选候选解；
- **状态抽象**：把复杂种群压缩为 ParetoState；
- **策略控制**：用规则或 LLM 对搜索参数做在线干预；
- **经验记忆**：记录状态-动作-反馈链条；
- **实验评估**：支持 baseline、消融、指标导出和结果比较。

如果把它看成一个研究平台，那么当前仓库已经具备一个相当扎实的原型框架；如果把它看成一个持续演进的工程，那么最有价值的方向就是继续提升：

- 文档质量；
- 注释质量；
- 配置规范；
- 指标解释性；
- 实验严谨性。

这份文档的目标，就是帮助后来者更快理解这些设计意图，而不只是“知道怎么运行”。
